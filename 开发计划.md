# 开发计划

> 本文件是 MVP（V1）的施工蓝图。9 个任务 + 依赖关系 + 验收标准 + 风险应对。所有任务详情见对应 TaskList，本文件给"全局视图"。

---

## 1. 总体目标

完成 MVP（V1）：用户用一行命令安装后，可在任意学习目录通过 5 个 skill 完整走完学习闭环：

```
注入目标 → 制定计划 → 写理解 → 自动批注 → 写应用 → 自动批注 → 自动出题 → 答题 → 自动评判
```

## 4. 9 个任务详解（与 TaskList 一一对应）

### Task #1 创建目录骨架
- **目标**：建好 `skills/` 及 5 个子目录，含空 SKILL.md 占位
- **输出**：6 个目录 + 5 个空文件 + 1 个 `templates/` 子目录
- **验收**：`ls skills/` 显示 5 个 skill 目录；每个目录里有 `SKILL.md`
- **依赖**：无

### Task #2 学习信息注入/SKILL.md
- **目标**：第 1 个 skill（入口任务型）
- **输入**：技术细则.md §5.1（frontmatter + body 已写好，直接照搬）
- **输出**：`skills/学习信息注入/SKILL.md` <100 行
- **验收**：frontmatter 含 `disable-model-invocation: true`；body 含 5 块（何时使用 / 操作流程 / 输出模板 / 契约 / 注意事项）
- **依赖**：#1

### Task #3 学习计划制定/SKILL.md + 模板
- **目标**：第 2 个 skill（编排核心），同时生成笔记骨架
- **输入**：技术细则.md §5.2 + §3 笔记 schema
- **输出**：
  - `skills/学习计划制定/SKILL.md` <200 行
  - `skills/学习计划制定/templates/学习计划模板.md`
- **验收**：
  - frontmatter 含 `context: fork` + `agent: general-purpose`
  - body 明确说明"生成 学习笔记.md 骨架时按 §3 的 5 个锚点结构"
  - 模板文件可被 body 通过 `./templates/学习计划模板.md` 引用
- **依赖**：#1

### Task #4 理解批注/SKILL.md
- **目标**：第 3 个 skill（响应参考型）
- **输入**：技术细则.md §5.3 + §4.1 大文件读取规范
- **输出**：`skills/理解批注/SKILL.md` <150 行
- **验收**：
  - frontmatter 不加 `disable-model-invocation`，`allowed-tools` 只到 `Edit`（不要 `Write`）
  - body 明确"先 Grep 定位章节再 Read 局部"
  - body 明确"批注用 ✅⚠️❌💡 符号"
- **依赖**：#1

### Task #5 应用批注/SKILL.md
- **目标**：第 4 个 skill（响应参考型）
- **输入**：技术细则.md §5.4
- **输出**：`skills/应用批注/SKILL.md` <150 行
- **验收**：
  - 与理解批注的区别明确：必须读 `学习目标.md` 中的项目背景
  - body 含"三方对比"（用户应用 ↔ 资料标准做法 ↔ 用户实际项目）
- **依赖**：#1

### Task #6 出题/SKILL.md
- **目标**：第 5 个 skill（参考型，含出题+评判双分支）
- **输入**：技术细则.md §5.5
- **输出**：`skills/出题/SKILL.md` <200 行
- **验收**：
  - body 明确两个分支的触发条件互斥
  - 出题分支：3-5 道题、每题标注出题理由、自适应选型
  - 评判分支：只评判不出新题
  - **MVP 不加 `context: fork`**
- **依赖**：#1

### Task #7 install.sh
- **目标**：一行命令安装脚本
- **输出**：`install.sh`（bash，跨平台）
- **验收**：
  - 交互式问"装到项目级还是用户级"
  - 项目级 → 检查当前目录是否有 `.claude/`，没有则建
  - 用户级 → 检查 `~/.claude/skills/` 是否存在，没有则建
  - `cp -r skills/* <目标>/` 完成复制
  - 输出"安装成功，启动 Claude Code 即可使用 5 个 skill"
  - 失败时给出可读错误
- **依赖**：#2 #3 #4 #5 #6（要有 skill 内容才能安装）

### Task #8 README.md
- **目标**：用户文档
- **输出**：`README.md` ~150 行
- **验收**：包含
  - 一句话定位
  - 安装命令（`curl -sL <url>/install.sh | bash`，url 先用占位符）
  - 5 个 skill 简介表
  - 典型使用流程示例
  - 目录结构说明
  - V2 路线（plugin install）
- **依赖**：#7

### Task #9 端到端串联测试
- **目标**：用真实学习资料验证全流程
- **测试材料**：任意一份结构化 md 学习资料（放入学习目录的 `资料/` 下）
- **输入**：装好的 skill 包 + 测试材料
- **输出**：测试报告（口头/简短文档），含
  - 5 个 skill 触发情况（参考型是否在该触发时触发）
  - 笔记 schema 5 个锚点是否被正确读写
  - 大文件读取是否走 Grep+offset 路径
  - 出题 skill 实际 token 消耗（决定 V2 是否加 fork）
- **依赖**：#7 #8

## 7. 风险与应对

| 风险 | 概率 | 影响 | 应对 |
|---|---|---|---|
| 参考型 skill 触发不准（资料 14 讲：56% 不触发） | 高 | 高 | description 严格三段式 + 端到端测试专门验证触发率 |
| 学习笔记越写越大导致 token 爆 | 中 | 高 | 严格 §4.1 大文件读取规范；端到端测试时观察 |
| 出题 skill token 消耗超预期 | 中 | 中 | 端到端测试记录消耗，决定 V2 是否加 fork |
| install.sh 跨平台问题 | 中 | 中 | 主测 Windows Git Bash；其他平台靠反馈 |
| Claude Code 框架不识别中文 skill 名 | 低 | 高 | 开发前先在 #2 完成时立即手动测一次加载 |
| 5 个 skill 之间契约对接出错（如锚点拼写不一致） | 中 | 高 | 写 skill 时必须 copy-paste §3 的锚点字符串，不要手敲 |

---

## 8. 不在 MVP 范围（V2+ 才做）

- 发布到 plugin registry（`claude plugin install`）
- 给出题 skill 加 `context: fork`
- 各 skill 的 `examples/` 子文件（除非测试发现必要）
- 多语言（仅中文）
- 学习数据可视化 / 进度报告
- 多人协作功能

---

## 9. 开工建议

按依赖关系，推荐这个执行顺序：

1. **先做 #1 #2**（建骨架 + 第 1 个 skill），跑一次手动加载测试，验证框架能识别中文 skill 名
2. **如果加载 OK** → 串行做 #3-#6（按顺序保证一致性，每写完一个 grep 一下笔记锚点字符串确保拼写一致）
3. **如果加载有问题** → 立即停下来调整命名（避免 5 个 skill 都写完才发现要返工）
4. #7 #8 #9 按顺序做
