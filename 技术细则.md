# 技术细则 — 共享契约 + 6 个 Skill 设计稿

> 本文件是 skill 开发的**蓝图与硬约束**。第 1-4 节是跨 skill 的共享契约（所有 skill 必须遵守），第 5 节是每个 skill 的完整设计（含 frontmatter + body 主体）。开发时按本文件创建实际的 SKILL.md。
>
> **V1.1 升级要点**（2026-05-23）：(1) 笔记 schema 加第 6 锚点「学习总结」；(2) 字段占位改为引用块 + 标识符；(3) 批注加 HTML span 颜色；(4) 新增「学习总结」skill。

---

## 1. 协作方案（方案 C 混合型）

| Skill | 触发类型 | frontmatter 关键字段 | 触发时机 |
|---|---|---|---|
| 学习信息注入 | 任务型 | `disable-model-invocation: true` | 用户首次开始学习时手动 `/学习信息注入` |
| 学习计划制定 | 任务型 | `disable-model-invocation: true` | 档案+资料齐备时手动 `/学习计划制定` |
| 理解批注 | 参考型 | 不加禁用字段 | 用户在 `学习笔记.md` 写完「我的理解」时 Claude 自动触发 |
| 应用批注 | 参考型 | 不加禁用字段 | 用户写完「我的应用」时 Claude 自动触发 |
| 出题 | 参考型 | 不加禁用字段 | 两个批注都齐备 + 用户表达想做题时 Claude 自动触发 |
| **学习总结**（V1.1 新增）| 参考型 | 不加禁用字段 | 某章节所有答题都已评判 + 「学习总结」字段未填时 Claude 自动触发 |

**笔记生成**不独立成 skill，由 `学习计划制定` 在创建 `学习计划.md` 时一并生成 `学习笔记.md` 骨架。

## 3. 学习笔记 Schema（V1.1 升级 · 最关键契约）

`学习笔记.md` 骨架由 `学习计划制定` skill 按学习计划一次性生成所有章节。**每章使用以下固定结构**：

````markdown
# 学习笔记

## 第 1 章 [章节标题]

### 我的理解
> ✍️ **[请在此处写你的理解]** —— 写完保存后告诉 Claude

### 大模型批注（理解）
> 🤖 **[AUTO]** Claude 会在此处自动填批注，请勿编辑

### 我的应用
> ✍️ **[请在此处写你的应用方案]** —— 写完保存后告诉 Claude

### 大模型批注（应用）
> 🤖 **[AUTO]** Claude 会在此处自动填批注，请勿编辑

### 出题与答题
> 🤖 **[AUTO]** Claude 会在此处出题，你在每道题下方写答案

### 学习总结
> 🤖 **[AUTO]** 答题评判完成后，Claude 会在此处汇总本章学习要点

## 第 2 章 [章节标题]
（同上结构，6 个锚点不得改名）
````

### 6 个固定锚点（不得改名）
1. `### 我的理解`
2. `### 大模型批注（理解）`
3. `### 我的应用`
4. `### 大模型批注（应用）`
5. `### 出题与答题`
6. `### 学习总结` ← V1.1 新增

### 占位写法（V1.1）
- **用户字段**用 `> ✍️ **[请在此处写...]** —— 提示`，引用块格式让用户一眼看出"这里要写"
- **大模型字段**用 `> 🤖 **[AUTO]** 说明`，引用块格式让用户避免误写

### 状态判断规则（V1.1 升级）
**不再判断"字段是否为空"**，改用**占位标识符识别**：

| 字段类型 | 字段下含 | 判定为 |
|---|---|---|
| 用户字段（我的理解 / 我的应用 / 我的答案 N）| `[请在此处写` | **未填**，不要触发批注 skill |
| 用户字段 | 不含 `[请在此处写` | **已填**，可触发对应批注 skill |
| 大模型字段（4 个批注 / 出题与答题 / 学习总结）| `[AUTO]` | **未填**，是该 skill 的待办 |
| 大模型字段 | 不含 `[AUTO]` | **已填**，下游 skill 可依赖 |

### 异常输入识别
如果发现**用户在 `[AUTO]` 占位的位置写了内容**（即大模型字段下不含 `[AUTO]` 但也不像 Claude 生成的批注格式），按 §4.3 处理：提示用户写错位置，询问是否帮转移。

### 批注语言规范（V1.2 简化 · emoji-only）

所有批注用 emoji + 简短描述，**严禁包 HTML span 标签**（V1.1 曾用 `<span style="color:...">`，但在 VS Code 默认编辑视图、GitHub web 等场景不渲染，反而暴露 raw HTML 字符串干扰阅读）：

| 符号 | 用途 | 颜色（emoji 自带）|
|---|---|---|
| ✅ | 完全正确 / 到位 / 表扬 | 绿色 |
| ⚠️ | 部分正确 / 理解不完整 / 需强调 | 黄色 / 橙色 |
| ❌ | 错误（说明错在哪、正确是什么）| 红色 |
| 💡 | 拓展提示 / 没想到的关键点 | 黄色 |

**示例**（直接 emoji + 文字，无 HTML 标签）：

```
✅ 无状态的理解到位
❌ HTTP 与 TCP 混淆 —— HTTP 是应用层，TCP 是传输层
⚠️ "请求-响应"理解不完整：漏了 HTTP/2 多路复用
💡 没提到无状态如何配合 Cookie/Session 模拟会话
```

每条一行，避免长段落。

## 4. 公共操作规范（所有 skill 必须遵守）

### 4.1 大文件读取规范
`学习笔记.md` 会越写越长。读取时**必须**：
1. 先 `Grep "## 第 N 章" 学习笔记.md` 定位章节起始行号
2. 再 `Read` 时指定 `offset` 和 `limit`，只读当前章节范围
3. 禁止全文 Read（除非文件 <200 行）

### 4.2 契约式引用规范
SKILL.md body 引用其他文件时，必须写齐**三要素**：

```markdown
# ❌ 弱引用
详见 `templates/批注模板.md`

# ✅ 契约式引用
**当需要参考标准批注样式时**，加载 `templates/批注模板.md`
**内容**：4 类批注符号的标准用法 + 3 个完整示例
```

### 4.3 公共错误处理
所有 skill 在以下情况必须停止并提示用户，禁止"好心办坏事"：

| 异常 | 行为 |
|---|---|
| 前置文件缺失（如理解批注找不到 `学习笔记.md`） | 提示用户先调用对应前置 skill，本 skill 退出 |
| 学习资料找不到 | 询问用户资料路径，**禁止虚构资料内容** |
| 章节定位失败 | 提示"未找到章节 X，请检查 `学习笔记.md` 是否被手动改动"，退出 |
| 学习目标过于宽泛（如"学 Python"） | 提示用户回到 `学习信息注入` 细化目标到可衡量级别 |
| 用户字段仍含 `[请在此处写` 标识符 | 视为"未填"，提示"你还没写第 N 章「xxx」"，退出 |
| 用户在 `[AUTO]` 占位的大模型字段处写了内容 | 提示"你似乎把内容写错了位置 —— 此处是 Claude 填写区。正确位置是「我的理解 / 我的应用 / 我的答案 N」。需要我帮你把内容转移过去吗？" |

### 4.4 渐进式披露规范
每个 skill 的目录结构必须遵循：

```
<skill-name>/
├── SKILL.md           ← <500 行，仅含操作流程
├── templates/         ← 输出模板（被 skill 引用而非内联）
└── examples/          ← 示例样本（按需加载，可选）
```

SKILL.md body 超过 500 行必须拆。

### 4.5 权限最小化
`allowed-tools` 按需声明。参考型 skill 通常只需 `Edit`（不是 `Write`），因为它们修改已有文件而不创建新文件。

### 4.6 标识符 Grep 速查（V1.1 新增）
状态检查统一用以下 Grep 模式：

```bash
# 用户字段是否已填（已填 = 不含 [请在此处写）
grep -A 3 "### 我的理解" 学习笔记.md | grep -q "\[请在此处写"
# 退出码 0 = 还含 = 未填；非 0 = 不含 = 已填

# 大模型字段是否待办（待办 = 含 [AUTO]）
grep -A 3 "### 大模型批注（理解）" 学习笔记.md | grep -q "\[AUTO\]"
# 退出码 0 = 含 = 待办；非 0 = 不含 = 已完成
```

**注意**：Grep 工具用 `\[AUTO\]` 转义方括号；用 `-A 3` 看字段标题下 3 行。

name: 学习计划制定
description: 当用户已建好 学习目标.md、放入学习资料、准备开始系统学习时调用。对比学习目标与学习资料制定个性化学习计划，生成 学习计划.md，并按计划一次性创建 学习笔记.md 骨架（6 锚点 + 引用块占位）。

### 5.3 理解批注（参考型 · V1.1 状态检查 + 颜色升级）

frontmatter 不变。**V1.1 关键变更**：

1. **状态检查改用标识符**：
   - 触发条件 = 「我的理解」字段下**不含** `[请在此处写` + 「大模型批注（理解）」字段下**含** `[AUTO]`

3. **Edit 替换占位整行**：old_string 必须包含整行 `> 🤖 **[AUTO]** Claude 会在此处自动填批注，请勿编辑`，new_string 是生成的批注（不再含任何 `[AUTO]`）

### 5.5 出题（参考型 · V1.1 双分支同步升级）

frontmatter 不变。**V1.1 关键变更**：

**出题分支** Step 4 —— 题目骨架占位也用新格式：

````markdown
### 出题与答题

#### 题目 1（针对 xxx 批注）
[题目内容]

##### 我的答案 1
> ✍️ **[请在此处写你的答案]** —— 写完保存后告诉 Claude

##### 大模型批注 1
> 🤖 **[AUTO]** 待评判

#### 题目 2（针对 xxx 批注）
（同上结构）
````

Edit 写入「出题与答题」时：old_string 是整行 `> 🤖 **[AUTO]** Claude 会在此处出题...`，new_string 是题目骨架。

**评判分支**：
- 状态检查：「我的答案 N」下不含 `[请在此处写` + 「大模型批注 N」下含 `[AUTO]`
- 批注用新颜色规范
- 评判完后告知用户："✅ 第 N 章答题已评判（✅ X / ⚠️ Y / ❌ Z）。学习总结即将自动生成。"（**不再自己写总结，交给 [[学习总结]] skill**）

### 5.6 学习总结（V1.1 新增 · 参考型）

```yaml
---
name: 学习总结
description: 当用户某章节「出题与答题」下所有题目的「大模型批注 N」都已填（不含 [AUTO] 标识符）、且该章节「学习总结」字段含 [AUTO] 时自动触发。读取该章节前 5 部分（理解 / 理解批注 / 应用 / 应用批注 / 出题与答题），生成包含"已掌握 / 需重点复习 / 拓展建议 / 整体评估"四段的学习总结，写入「学习总结」字段。
allowed-tools:
  - Read
  - Grep
  - Glob
  - Edit
model: sonnet
---
```

**SKILL.md body**：

```markdown
# 学习总结

## 何时使用
**同时满足**：
- 存在 `学习笔记.md`
- 某章节「出题与答题」下所有「大模型批注 N」字段均已填（用 Grep 查所有 `##### 大模型批注` 行后无 `[AUTO]`）
- 同章节「学习总结」字段仍含 `[AUTO]`

不满足时不要触发。

## 操作流程

### Step 1 — 定位章节
按 §4.1 大文件读取规范定位目标章节（Grep + offset Read）。

### Step 2 — 读取该章节前 5 部分内容
仅读当前章节范围（用 offset/limit）：
- 我的理解
- 大模型批注（理解）
- 我的应用
- 大模型批注（应用）
- 出题与答题（含所有题目、答案、答案批注）

### Step 3 — 提取分类条目
从所有大模型批注中提取（包括 4 个章节批注 + 答题批注）：
- 所有 ✅ 条目 → 已掌握的方面
- 所有 ⚠️ ❌ 条目 + 答错的题 → 需重点复习的方面
- 所有 💡 条目 → 拓展建议

### Step 4 — 生成四段式总结
按以下结构，所有内容用 §3 批注颜色规范包裹：

```markdown
### 学习总结

**✅ 已掌握**
<span style="color:#22c55e">✅ [具体掌握点 1，引用对应批注]</span>
<span style="color:#22c55e">✅ [具体掌握点 2]</span>

**⚠️ 需重点复习**
<span style="color:#ef4444">⚠️ [薄弱点 1，给出复习建议]</span>
<span style="color:#ef4444">❌ [错误点，明确正确做法]</span>

**💡 拓展建议**
<span style="color:#3b82f6">💡 [拓展方向 1]</span>

**整体评估**
[一句话总结本章掌握程度，如"基础概念扎实，但应用场景理解不够深入，建议先复习 XX 再进入下一章。"]
```

### Step 5 — 写入「学习总结」
用 Edit 精确替换该字段下的整行 `> 🤖 **[AUTO]** 答题评判完成后，Claude 会在此处汇总本章学习要点` 为生成的总结。

### Step 6 — 告知用户
输出："✅ 第 N 章学习总结已生成。建议复习「需重点复习」部分后进入第 N+1 章。"

## 契约
- **读入**：`学习笔记.md`（仅该章节前 5 部分，按 §4.1 局部读）
- **写出**：`学习笔记.md`（仅对应章节「学习总结」字段）

## 大文件读取（强制）
同 [[理解批注]]，先 Grep 定位再 Read 局部。

## 注意事项
- **不要重新评判**：总结只是汇总，不要重新给批注或挑战已有评价
- **总结要具体**：不要说"理解整体到位"这种空话，必须引用具体的批注条目
- **四段必须齐全**：即使某段没内容也要写"（本章无）"以保持结构
- **整体评估一句话**：避免冗长，给用户清晰的"我学得怎么样"判断
- **不要触发出题或其他 skill**：本 skill 是学习闭环的终点
```

## 6. 命名规范

| 对象 | 命名规则 |
|---|---|
| skill 目录名 | 中文，与 `name` 字段一致（如 `学习信息注入/`） |
| frontmatter `name` | 中文动名词或动词短语 |
| frontmatter yaml key | 必须英文（Claude Code 框架要求） |
| 契约文件名 | 中文，与第 2 节命名约定一致 |
| skill 子目录 | 固定使用 `templates/`、`examples/`（英文，框架习惯） |
| body 中引用其他 skill | 用 `[[skill 名称]]` 双链格式（如 `[[学习计划制定]]`） |
